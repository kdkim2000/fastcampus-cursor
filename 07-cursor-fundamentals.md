# Chapter 7. Cursor란 무엇인가

앞선 챕터들에서 우리는
- Agent 개념
- Context Engineering
- AI 활용 전략과 보안
을 살펴보았다.

이제 그 개념들이
**실제로 구현된 도구가 무엇인지** 를 살펴볼 차례이다.
그 도구가 바로 **Cursor** 이다.

---

## 7.1 Cursor의 포지션

### 7.1.1 IDE 기반 AI 서비스

Cursor는 단순한 AI 채팅 도구가 아니다.

> **Cursor는 IDE 기반 AI 서비스** 이다.

이는 매우 중요한 차이점이다.

- 코드 옆에서 AI가 동작한다
- 프로젝트 구조를 그대로 인식한다
- 파일, 디렉토리, 변경 이력을 맥락으로 사용한다

기존 방식:
- IDE ↔ AI 도구 (분리)

Cursor 방식:
- **IDE = AI 실행 환경**

👉 AI가 “외부 도구”가 아니라  
개발 환경 그 자체에 내장되어 있다.

---

### 7.1.2 Cursor의 위치 정리

Cursor는 다음 영역의 교차점에 위치한다.

- IDE
- Code Assistant
- Agent 실행기

즉, Cursor는:
- 단순 자동완성 도구가 아니라
- **Agent를 실행하기 위한 플랫폼** 이다.

---

## 7.2 왜 Cursor인가

AI 코딩 도구는 많지만,
왜 이 강의에서 Cursor를 중심으로 다루는지에는
명확한 이유가 있다.

---

### 7.2.1 시각적 Context

Cursor의 가장 큰 강점 중 하나는
**Context를 ‘보여준다’는 점** 이다.

- 어떤 파일이 Context에 포함되었는지
- 어떤 변경이 발생했는지
- AI가 무엇을 참고하고 있는지

이 모든 것이
IDE 화면 안에서 시각적으로 확인 가능하다.

👉 이는 개발자에게 다음과 같은 이점을 준다.

- 통제감 유지
- 예측 가능성 증가
- 실수 및 오작동 감소

---

### 7.2.2 Agent 중심 설계

Cursor는 처음부터
**Agent 사용을 전제로 설계된 도구** 이다.

- 단일 명령이 아닌 작업 단위 실행
- 여러 파일을 탐색하고 수정
- 반복 실행과 피드백 루프 지원

즉, Cursor는:
- “코드를 도와주는 도구”가 아니라
- **“일을 대신 수행하는 도구”** 이다.

---

## 7.3 Context 생애주기

Cursor를 제대로 사용하기 위해서는
Context를 “한 번 주고 끝”이라고 생각하면 안 된다.

Context에는 **생애주기(Lifecycle)** 가 존재한다.

> 생성 → 유지 → 압축

---

### 7.3.1 Context 생성

Context 생성 단계에서는
Agent가 작업을 수행하기 위해
필요한 정보를 제공한다.

- 목표(Goal)
- 관련 파일
- 도메인 설명
- 제약 조건

이 단계가 부실하면,
Agent의 결과물도 부실해진다.

👉 Context 품질 = 결과물 품질

---

### 7.3.2 Context 유지

작업이 진행되면서
Context는 계속 변화한다.

- 파일 변경
- 추가 설명
- 새로운 제약 조건

Cursor는:
- 현재 상태를 시각적으로 보여주고
- Context가 어떻게 확장되고 있는지 인지하게 해준다.

👉 유지 단계에서는
**불필요한 Context를 늘리지 않는 것이 중요** 하다.

---

### 7.3.3 Context 압축

작업이 끝난 후에는
Context를 정리해야 한다.

- 핵심 규칙 요약
- 반복 설명 제거
- 다음 작업에 필요한 정보만 남김

이를 **Context 압축(Compression)** 이라고 한다.

압축하지 않으면:
- 컨텍스트 과부하
- 성능 저하
- 혼선 발생

👉 좋은 Cursor 사용자는
**Context를 관리하는 사람** 이다.

---

## 7.4 Chapter 7 정리

- Cursor는 IDE 기반 AI 서비스이다
- Cursor는 Agent 실행 플랫폼이다
- 시각적 Context는 통제력을 높인다
- Cursor는 Agent 중심으로 설계되었다
- Context에는 생성–유지–압축의 생애주기가 있다
