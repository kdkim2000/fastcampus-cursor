## 개발자에게 AI 도구의 의미

### AI 개발 도구의 등장

1. **규칙 기반의 시작: Linter의 등장**
   - 정해진 규칙으로 오류 검출
   - 코드 스타일 일관성 확보

2. **의미론적 분석의 시대: 정적 분석 도구 고도화**
   - 코드의 구조와 의미를 분석
   - 품질, 보안 등 심층 분석

3. **패러다임의 전환: 코딩 어시스턴트 등장**
   - '분석'에서 '생성'으로 전환
   - AI가 코드를 직접 생성

4. **자율성을 향하여: AI 에이전트의 부상**
   - 스스로 계획하고 실행
   - '조수'에서 '파트너'로 진화

### Lint의 등장

- "lint"라는 용어는 옷에서 나오는 보풀을 의미하며 정적분석이 이러한 코드의 보풀을 제거한다는 관점에서 시작됨
- 1970년대 후반 C언어용으로 개발된 lint는 코드 스타일 오류 및 잠재적인 버그를 식별하는데 활용됨
- 현재도 Type기반의 언어에서 lint가 적극적으로 활용되고 있음 (Typescript, Java, C 등)

### 의미론적 분석의 시작

- 2000년 이후 등장한 3세대 도구들은 AST(Abstract Syntax Tree) 모델링을 통해 코드의 의미론적 분석 능력을 크게 향상시킴
- 우리에게도 익숙한 SonarQube와 같은 도구들이 3세대에 속하며, 코드 품질, 보안, 유지보수성 등 광범위한 측면을 분석할 수 있게됨
- 미리 정의된 규칙에 따라 작동하며, 제한된 범위 내에서 특정 작업을 수행할 수 있었지만, **유연성이 부족함**

## VIBE Coding이란?

2025년 2월, OpenAI 공동창업자 Andrej Karpathy가 올린 글에서 "완전히 바이브에 몸을 맡기고, 지속적 성장을 받아들이며, 코드가 존재한다는 것조차 잊어버리는"이라는 문장에서 처음으로 **Vibe Coding**이라는 문구가 등장했습니다.

## Vibe Coding의 진행 방식

1. **개발자가 의도를 파악**
2. **자연어로 의도를 설명**
3. **AI가 의도에 맞게 코드를 생성**
4. **개발자는 코드를 그대로 수용**


## 코드의 '맥락'을 모른다

코드에 숨겨진 의도와 배경을 모르기에 레거시 코드 수정이 어렵다.

```javascript
export function initialize() {
    if (window.performance.timing.domComplete) {
        // 페이지 로딩이 끝났으면 'ready' 상태로 설정
        window.app_status = 'ready';
    } else {
        // 아직 로딩 중이면 'loading' 상태로 설정
        window.app_status = 'loading';
    }
}
```

## 개발의 맥락은 개발자의 경험에 녹아있습니다

- 코드가 다른 코드와 결합되는가? 아키텍처 구조상 필요한 코드인가?
- 작성 당시의 도메인 상황에 부합한가? 특정 도메인에서의 예외조항이 있는가?
- 타 부서와의 협업상 필요한 코드인가? 코드에 안보이는 조직의 규칙이 있는가?
- 데이터 구조상의 예외상황이 있는가? 특정 용도를 위한 데이터인가?

## Vibe Coding은 코드의 의도를 모르고 작업에 들어간다

- 코드가 작성될 당시의 상황을 알고 있는가?
- 코드가 미치는 영향을 알고 있는가?
- 내가 전달한 지시사항이 내 모든 의도를 담고 있는가?
