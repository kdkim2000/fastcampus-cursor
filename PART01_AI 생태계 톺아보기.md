# Part 1. AI 생태계 톺아보기
## 강의개요

이 파트에서는 단순히 “AI 도구를 사용하는 법”이 아니라,  
**개발자가 AI와 함께 일하는 방식 자체가 어떻게 바뀌고 있는지** 를 다룹니다.

과거의 AI는 질문에 답해주는 도구에 가까웠다면,  
이제 AI는 **일을 맡길 수 있는 주체(Agent)** 로 진화하고 있습니다.

이 변화의 흐름을 이해하지 못하면,
- 왜 실무에서 AI 활용이 어설퍼지는지
- 왜 데모에서는 잘 되는데 프로젝트에서는 실패하는지
- 왜 “Vibe Coding”이 불안정한 결과를 만드는지

그 이유를 설명할 수 없습니다.

Part 1은 이후 Cursor, Agent, MCP, Workflow를 **왜 이렇게 사용해야 하는지** 에 대한
사고의 토대를 만드는 장입니다.

---

### 개발자에게 AI 도구의 의미

AI 도구는 더 이상 “코드를 대신 써주는 자동완성기”가 아닙니다.  
개발자에게 AI의 의미는 다음 단계로 이동했습니다.

#### 1️⃣ 코드 어시스턴트(Code Assistant)의 시대

초기의 AI 코딩 도구는 **보조자(Assistant)** 역할에 가까웠습니다.

- Github Copilot
- 초기 Cursor의 Tab / Chat 기능
- 단일 파일 또는 짧은 코드 맥락 기반 제안

이 시기의 AI는:
- 개발자가 작성 중인 **코드 일부만 이해**
- 질문에 답하거나, 코드 한두 줄을 제안
- “페어 프로그래머”에 가까운 역할

즉, **개발의 주체는 여전히 사람** 이었고  
AI는 그 옆에서 힌트를 주는 존재였습니다.

#### 2️⃣ 개발자가 느낀 한계

실무에서는 곧 한계가 드러났습니다.

- 파일이 많아질수록 맥락을 잃음
- 프로젝트 구조를 이해하지 못함
- 반복 작업은 여전히 사람이 직접 수행
- “왜 이렇게 작성했는지”의 의도를 이해하지 못함

이로 인해 개발자들은 점점 이렇게 느끼게 됩니다.

> “도움은 되는데, 일을 맡기기에는 불안하다”

---

### 왜 Vibe Coding을 실무에서 하기 어려운가

Vibe Coding은 “느낌대로 AI에게 맡기면 코드가 나온다”는 접근입니다.  
데모나 개인 프로젝트에서는 꽤 그럴듯하게 작동합니다.

하지만 실무에서는 거의 항상 문제가 됩니다.

#### ❌ Vibe Coding의 구조적 한계

1. **맥락이 정의되지 않음**
   - 프로젝트 목적
   - 팀 규칙
   - 기술 스택의 제약

2. **결과 기준이 없음**
   - “잘 동작하면 OK”
   - 테스트, 유지보수, 확장성 고려 부족

3. **책임의 주체가 불분명**
   - AI가 만든 코드의 품질을 누가 보증하는가?

결국 Vibe Coding은  
> “우연히 잘 되면 좋은 코드”

에 의존하는 방식입니다.

실무 개발은 우연이 아니라 **재현 가능성** 이 핵심입니다.

---

### Chat에서 Agent로

AI 활용의 패러다임은 **Chat → Agent** 로 이동하고 있습니다.

#### Chat 기반 협업의 특징

- 질문 ↔ 답변의 반복
- 문제를 함께 정의하고 토론
- 불확실성이 큰 영역에 적합

예:
- “JWT랑 세션 중 뭐가 좋을까?”
- “이 구조에서 확장성 문제는 없을까?”

이는 **협업(Collaboration)** 에 해당합니다.

#### Agent 기반 위임의 특징

Agent는 단순 대화 상대가 아닙니다.

- 목표를 받고
- 계획을 수립하고
- 필요한 도구를 사용하며
- 결과물을 만들어냅니다

즉, **일을 맡길 수 있는 존재** 입니다.

이를 위해 필요한 것이 바로:

1. **Workflow**
   - 내가 일을 처리하는 방식의 구조화

2. **Instruction**
   - Agent가 따라야 할 규칙과 기준

3. **Context**
   - 코드, 문서, 규칙, 제약조건의 묶음

#### Assistant와 Agent의 결정적 차이

| 구분 | Code Assistant | Code Agent |
|----|----|----|
| 역할 | 보조자 | 독립적인 작업 주체 |
| 맥락 | 제한적 (현재 파일 중심) | 전체 프로젝트 |
| 작업 방식 | 제안 | 계획 → 실행 |
| 책임 | 사람 | 사람 + Agent |

---

### 이 강의에서 다루는 핵심 관점

이 강의는 다음 질문에 답하는 것을 목표로 합니다.

- AI에게 **어디까지 맡길 수 있는가?**
- AI가 **내 방식대로 일하게 하려면 무엇이 필요한가?**
- Chat을 넘어서 **Agent를 어떻게 설계하는가?**

Part 1에서는 이 질문에 대한 **개념적 기반** 을 다지고,  
Part 2와 Part 3에서 Cursor와 Agent를 통해 **실제로 구현** 해봅니다.

---

### 정리

- AI 도구는 단순 보조가 아니라 **협업·위임의 대상** 이 되고 있다
- Vibe Coding은 실무에서 재현 가능하지 않다
- Chat에서 Agent로의 전환이 핵심 변화다
- 개발자의 역할은 “작성자”에서 **설계자·감독자** 로 이동하고 있다

## 개발에 필요한 AI 딥다이브

이 섹션에서는 “AI를 어떻게 쓰는가”보다 한 단계 더 들어가  
**개발자가 AI를 어떤 역할로 대해야 하는가** 를 다룹니다.

같은 AI라도
- 어떤 도구를 선택하느냐
- 어떤 맥락을 주느냐
- 협업으로 쓸지, 위임으로 쓸지

에 따라 결과는 극단적으로 달라집니다.

---

### 개발에 활용 가능한 AI 도구의 종류

AI 도구는 모두 같은 역할을 하지 않습니다.  
**역할별로 구분해서 이해해야 실무에서 올바르게 사용할 수 있습니다.**

#### 1️⃣ 범용 대화형 AI – Claude

**Claude** 는 Anthropic에서 만든 LLM 기반 AI로,  
개발자에게는 “가장 신뢰할 수 있는 사고 파트너”에 가깝습니다.

- 강점
  - 코드 이해력 우수
  - 긴 맥락(Context) 처리
  - 설계·리뷰·토론에 강함

- 적합한 용도
  - 설계 논의
  - 코드 리뷰
  - 복잡한 요구사항 정리
  - 문서 요약 및 재구성

Claude는 **협업(Collaboration)** 에 가장 적합한 도구입니다.

---

#### 2️⃣ IDE 기반 AI Coding Tool – Cursor

**Cursor** 는 VS Code 기반의 AI Coding IDE입니다.  
단순 자동완성이 아니라, **프로젝트 단위로 일을 맡길 수 있는 환경** 을 제공합니다.

주요 기능:

- **Tab**
  - 현재 맥락 기반 코드 자동 완성

- **Inline Edit**
  - 특정 코드 블록에 대한 즉각적인 수정

- **Agent**
  - 전체 Codebase를 탐색
  - Rules를 기반으로 계획 수립
  - Tool(MCP)을 호출하며 작업 수행

Cursor는 이 강의의 핵심 도구이며,  
**“AI에게 개발을 위임한다”는 개념을 가장 잘 체험할 수 있는 환경** 입니다.

---

#### 3️⃣ 프론트엔드 자동화 도구 – V0

**V0** 는 Vercel에서 제공하는 React 기반 UI 자동 생성 도구입니다.

- 자연어로 UI 요구사항 입력
- React 컴포넌트 코드 생성
- 실시간 미리보기 제공

V0는:
- 빠른 프로토타이핑
- UI 시안 검증
- 프론트엔드 초기 생산성

에 매우 적합하지만,  
**비즈니스 로직이나 아키텍처를 맡기기에는 한계** 가 있습니다.

---

#### 4️⃣ AI Code Review 도구 – CodeRabbit

**CodeRabbit** 은 Pull Request 단위로 동작하는 AI Code Review Agent입니다.

- PR의 맥락 파악
- 코드 개선점 제안
- 잠재적 버그 지적

특히 조직 단위에서:
- 리뷰 품질의 최소 기준 유지
- 리뷰 속도 향상
- 시니어 의존도 감소

에 큰 도움이 됩니다.

### 맥락(Context)을 어떻게 전달해야 하는가

AI가 똑똑하지 않은 이유는 거의 없습니다.  
대부분의 실패는 **맥락 전달 실패** 입니다.

#### Context란 무엇인가?

Context는 단순한 “설명”이 아니라  
**AI가 판단하고 행동하기 위해 필요한 정보 묶음** 입니다.

Context는 보통 다음 요소로 구성됩니다.

- **목표(Goal)**  
  무엇을 해야 하는가?

- **지식(Knowledge)**  
  관련 코드, 문서, API

- **제약조건(Constraints)**  
  규칙, 컨벤션, 보안 정책

이 세 가지가 빠지면,  
AI는 항상 “그럴듯하지만 어딘가 어긋난 결과”를 만듭니다.

---

#### AI를 신입사원이라고 생각해보자

AI는 매우 유능하지만, 특징이 하나 있습니다.

> **기억력(Context Window)에 한계가 있다**

그래서 항상 다음을 묻습니다.

- 어떤 파일을 보면 될까요?
- 지금은 어디까지 신경 쓰면 되나요?
- 우리 팀 규칙은 어디 있나요?

이 질문에 답하지 않으면,  
AI는 **추측으로 일하게 됩니다.**

---

#### Context Engineering – 4가지 핵심 전략

실무에서 검증된 전략은 다음 네 가지입니다.

1. **Select (선택)**
   - 필요한 정보만 명확히 지정
   - “이 두 파일만 먼저 보세요”

2. **Isolate (격리)**
   - 지금 필요 없는 맥락은 배제
   - “인프라는 지금 신경 쓰지 마세요”

3. **Write (기록)**
   - 규칙과 기준을 문서로 남김
   - Rules, Wiki, Markdown

4. **Compress (압축)**
   - 긴 맥락을 요약하여 전달
   - Context Window 관리

이 전략은 이후 **Rules / MCP / Agent 설계의 핵심 기반** 이 됩니다.

---

### AI를 위임과 협업의 관점으로 바라보기

AI와 일하는 방식은 크게 두 가지입니다.

#### 협업 (Collaboration)

- 불확실성이 높은 단계
- 설계, 토론, 문제 정의
- 질문 ↔ 답변 중심

예:
- “JWT와 세션 중 어떤 게 나을까?”
- “이 구조에서 확장성 문제는 없을까?”

---

#### 위임 (Delegation)

- 목표와 기준이 명확한 단계
- 반복 가능하고 구조화된 작업
- 명령 → 실행 중심

예:
- “이 규칙에 맞춰 로그인 API 구현해줘”
- “이 Workflow대로 테스트 코드 작성해줘”

👉 **협업 없이 바로 위임하면 실패하고,  
위임 없이 계속 협업하면 생산성이 떨어집니다.**

---

#### 실무에서의 이상적인 흐름

1. 협업 – 문제 정의
2. 협업 – 설계 및 기준 합의
3. 위임 – 구현
4. 협업 – 결과 검토 및 개선

이 흐름을 **Agent 기반 개발** 로 자동화하는 것이
이 강의의 핵심 목표입니다.

---

### Code Agent의 등장이 시사하는 것

2025년을 기점으로 AI는 명확히 한 단계 진화했습니다.

> “에이전트는 당신을 대신해서  
> 독립적으로 작업을 수행하는 시스템입니다.”

(Code Agent에 대한 OpenAI 정의)

#### Code Agent의 특징

- 목표 기반 작업 수행
- 계획 수립 → 실행 → 피드백
- Tool을 활용한 외부 시스템 연동
- 사람의 개입 없이도 일정 수준의 완결성 확보

대표적인 예:
- Claude Code
- Cursor Agent
- Codex, Gemini CLI

---

#### 개발자의 역할은 어떻게 바뀌는가?

| 과거 | AI 시대 |
|----|----|
| 코드 작성자 | 업무 설계자 |
| 손으로 구현 | 기준과 규칙 정의 |
| 모든 작업 직접 수행 | 위임 + 검토 |

개발자는 더 이상
“모든 코드를 직접 치는 사람”이 아니라  
**AI가 잘 일하도록 환경을 설계하는 사람** 이 됩니다.

---

### 정리

- AI 도구는 역할에 따라 구분해서 사용해야 한다
- 맥락(Context)은 결과의 품질을 결정한다
- 협업과 위임을 구분해야 한다
- Code Agent의 등장은 개발자의 역할 자체를 바꾸고 있다
